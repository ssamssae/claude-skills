# 작업일지 작성 & 깃허브 푸시

**지정 날짜에 수행한 모든 작업**(현재 프로젝트 + 다른 프로젝트 + 다른 Claude Code 세션)을 수집해 작업일지를 작성하고 GitHub에 올립니다. 단일 프로젝트만 기록하던 과거 동작과 다르게, 그날 하루 전체 궤적을 한 문서에 합칩니다.

## 절차

1. **날짜 결정**
   - `$ARGUMENTS`로 날짜(`YYYY-MM-DD`)가 주어졌으면 그 날짜 사용
   - 없으면 오늘 날짜 (KST): `date +%Y-%m-%d`
   - 이후 단계에서 이 날짜를 `TARGET_DATE` 로 지칭

2. **해당 날짜의 모든 Claude Code 세션 수집 (크로스 세션)**
   - 세션 저장 루트: `~/.claude/projects/*/` (각 프로젝트 폴더마다 세션 `.jsonl` 파일들이 있음)
   - `TARGET_DATE` 에 **활동이 있었던** 세션 파일 찾기:
     ```
     find ~/.claude/projects -maxdepth 2 -name "*.jsonl" -newermt "TARGET_DATE 00:00" ! -newermt "TARGET_DATE 23:59:59"
     ```
     또는 파일 내부 `timestamp` 필드(UTC ISO)가 TARGET_DATE(KST) 에 해당하는 라인만 골라도 됨. mtime 과 실제 대화 시간이 어긋날 수 있으면 후자가 더 정확.
   - 각 `.jsonl` 은 한 줄당 하나의 이벤트(JSON). 주요 타입: `user`, `assistant`, `tool_use`, `tool_result`, `file-history-snapshot` 등.
   - 추출 대상:
     - `type == "user"` 의 사용자 입력 텍스트(슬래시 커맨드 포함) — **"무엇을 요청했는지"**
     - `type == "assistant"` 의 응답 요약 — **"무엇을 답/실행했는지"** (툴콜 인자에서 파일 경로, Bash 명령 등 실제 작업의 증거)
   - 프로젝트 폴더 이름(`-Users-user-dutch-pay-calculator` 등) 을 보면 어느 프로젝트/어느 위치(`cwd`)에서 작업했는지 파악 가능. 이벤트 내부 `cwd` 필드도 참고.
   - 구분하여 수집: 앱 개발(예: dutch_pay_calculator, yakmukja, simple_memo_app), 홈 세션(`-Users-user`, 일반 질문/설정/텔레그램/도우미 성격), 기타.
   - 용량이 큰 세션은 `head`/`tail` 로 샘플링하지 말고 필요한 필드만 파이썬 원라이너로 파싱해 요약 추출(토큰 절약).

3. **모든 프로젝트의 그 날짜 git 커밋 수집**
   - 관심 프로젝트 후보: 2단계에서 잡힌 각 프로젝트 폴더의 실제 경로들(`~/dutch_pay_calculator`, `~/yakmukja`, `~/simple_memo_app`, `~/babmeokja`, `~/daejong-page`, `~/Projects/lonely-death-prevention` 등) + 현재 cwd
   - 각 프로젝트에서:
     ```
     git -C <repo> log --since="TARGET_DATE 00:00" --until="TARGET_DATE 23:59" --pretty=format:"%h %s" --all
     git -C <repo> branch --show-current
     ```
   - git 저장소가 아닌 폴더는 건너뜀(오류 무시)

4. **사용자가 수동 보강한 내용 병합**
   - 현재 대화 컨텍스트에서 사용자가 "오늘 OO도 했어", "어제 웹에서 OO 했어" 같은 내용을 얘기했으면 포함
   - `~/.claude/projects/-Users-user/memory/` 의 project 타입 메모리 중 TARGET_DATE 관련 항목이 있으면 참고
   - Claude 웹(claude.ai), Claude Cowork 스케줄 등 **로컬에 로그가 없는 활동**은 자동 수집 불가. 사용자가 명시적으로 알려준 경우에만 포함.

5. **작업 내용 요약**
   - 2~4단계에서 모은 모든 자료를 합쳐 주제별로 정리 (앱 단위로 묶어도 되고, 영역별—UI/기능/버그/최적화/배포/설정/학습·대화—로 묶어도 됨. 그날 성격에 맞게 판단)
   - 단순 질문·잡담은 의미 있는 것만 추려 "학습/상담" 단락으로 간단히 (예: "OOO 봇 수익화 전략을 정리했다")
   - 커밋 메시지만이 아니라 대화에서 드러난 맥락/의도까지 포함
   - 절대 비밀번호/토큰/API key 포함 금지. 내부 경로 중 `~/.claude` 하위의 세션 ID 같은 건 쓰지 않기.

6. **파일 작성 (버전 스냅샷 방식)**

   - 저장 위치 결정: 현재 cwd 가 어느 git 저장소 안이면 그 저장소의 `docs/worklog/`. 저장소 밖(예: `~/`)에서 실행 중이면 "메인" 저장소 한 곳을 선택 — 기본값은 그날 커밋이 가장 많은 프로젝트, 동률이면 사용자에게 한 줄로 확인 요청.
   - 폴더가 없으면 `mkdir -p docs/worklog`

   - **파일명 규칙 (앱스토어 버전처럼 누적 스냅샷)**: 같은 날짜에 여러 번 /worklog 를 돌려도 **이전 파일은 절대 수정·삭제하지 않고** 새 버전 파일을 생성한다.
     - 파일명 패턴: `YYYY-MM-DD_vX.Y.Z.md` (예: `2026-04-15_v1.0.0.md`, `2026-04-15_v1.0.1.md`, …)
     - 버전 결정: `ls docs/worklog/ | grep "^YYYY-MM-DD_v" | sort -V | tail -1` 의 결과가
       - 없으면 `YYYY-MM-DD_v1.0.0.md` 로 시작
       - 있으면(예: `_v1.0.2.md`) 패치 버전 +1 한 **새 파일** `_v1.0.3.md` 생성. 기존 파일은 손대지 않음.
     - **절대 기존 버전 파일을 Edit 하지 말 것.** 항상 Write 로 새 파일을 만든다. 기존 파일이 바뀌면 그 스냅샷이 사라져 히스토리 취지가 무너진다.
     - 이전 버전이 있으면 Read 로 읽어 맥락을 파악한 뒤, **직전 버전 이후 달라진 작업을 중심으로** 새 스냅샷을 작성한다. 누적 스냅샷 성격이므로 전날까지의 주요 흐름은 간단히 요약해 포함하되, 초점은 "이번 버전에서 새로 한 일".
     - 기존에 구 형식 `docs/worklog/YYYY-MM-DD.md` (버전 없음) 이 있으면 **그대로 두고**, 버전 파일은 `v1.0.0` 부터 새로 시작한다.

   - **파일 형식도 8번의 "산문체" 규칙을 그대로 따름**. 채팅 응답과 파일 본문은 동일.
   - 섹션 헤딩, 불릿, 마크다운 기호 없이 단락 단위 평서체로 작성
   - 파일 첫 줄은 `YYYY.MM.DD 작업일지 vX.Y.Z`

7. **커밋 & 푸시 (원본 저장소)**
   - **새로 만든 버전 파일만** stage: `git -C <worklog-repo> add docs/worklog/YYYY-MM-DD_vX.Y.Z.md`
   - 이전 버전 파일은 건드리지 않으므로 자동으로 스냅샷이 보존된다. 혹시 `git status` 에 다른 파일이 딸려 올라오면 분리해서 커밋.
   - 커밋 메시지:
     - `docs: YYYY-MM-DD 작업일지 vX.Y.Z` (간결하게 버전 명시)
   - 해당 저장소의 현재 브랜치로 push: `git -C <worklog-repo> push origin $(git -C <worklog-repo> branch --show-current)`

7b. **daejong-page(개인 홈페이지) 에도 동시 동기화**

   개인 홈페이지 https://ssamssae.github.io/daejong-page/worklog.html 가 작업일지를 공개 열람한다. 새 버전 작업일지가 만들어질 때마다 여기에도 동시 반영해야 한다.

   - 저장 경로:
     - 파일 복사: `~/daejong-page/worklog/YYYY-MM-DD_vX.Y.Z.md` (원본과 동일한 파일명, 내용 그대로)
     - 인덱스 업데이트: `~/daejong-page/worklog/index.json` 의 `entries` 배열에 해당 파일의 메타데이터 1건 추가
   - index.json entry 형식:
     ```
     {
       "file": "YYYY-MM-DD_vX.Y.Z.md",
       "date": "YYYY-MM-DD",
       "version": "vX.Y.Z",
       "title": "<작업일지 첫 문단 요약 1문장, 20~40자>",
       "updated": "<ISO 8601 KST>"
     }
     ```
   - **중복 방지**: index.json 에 이미 같은 `file` 필드가 있으면 그 entry 를 갱신만 한다(업데이트 타임스탬프, title). 중복 추가 금지.
   - 파일명이 포함하는 민감 정보(세션 ID, 경로 등) 가 있으면 파일 내용도 다시 한번 필터링해서 복사. 원본과 다른 공개본이 필요하면 `_public` 접미사는 붙이지 말고 동일 파일명 유지.
   - daejong-page 저장소에서도 git add + commit + push:
     - `git -C ~/daejong-page add worklog/YYYY-MM-DD_vX.Y.Z.md worklog/index.json`
     - 커밋 메시지: `docs: 작업일지 YYYY-MM-DD vX.Y.Z 동기화`
     - push: `git -C ~/daejong-page push origin main`
   - **daejong-page 의 /vA.B.C/ 스냅샷 폴더는 절대 건드리지 말 것.** 이 폴더들은 홈페이지 버전 스냅샷 용이며 worklog 자동 동기화는 루트 `worklog/` 폴더에만 반영한다. 홈페이지 자체의 버전업은 별도 작업.
   - 원본 저장소 push 성공 후 daejong-page 동기화 실패 시, 원본 커밋은 그대로 두고 사용자에게 한 줄로 보고("홈페이지 동기화 실패, 원본 저장은 완료").

8. **결과 보고 (네이버 카페 모바일 앱 안전 포맷, "산문체")**
   - 작업일지 **전체 본문**을 응답으로 보내야 함. 사용자가 바로 복사해 네이버 카페 모바일 앱에 붙여넣어도 에러 없이 들어가야 함.
   - 텔레그램 세션이면 reply 도구로 전송.

   ### 포맷 원칙: "그날 일기처럼, 산문체로"
   - 불릿 나열 금지. 문장을 자연스럽게 이어 쓸 것
   - 섹션은 한 문장 또는 짧은 한 줄로 시작 (예: "기능적으로는 복수 선택 삭제를 새로 넣었다.")
   - 섹션 사이는 빈 줄 1줄
   - 전체 길이는 대략 800~1500자 내 (너무 길면 카페 앱 붙여넣기 버벅)
   - 문어체보다 담담한 평서체 선호 ("~했다", "~만들었다", "~마쳤다")

   ### 금지 문자/기호
   - 이모지/이모티콘 전부 금지
   - 대괄호 `[]`, 꺾쇠 `<>`, 중괄호 `{}` 금지
   - 화살표 `→`, `->` 금지. "에서", "로", "후" 같은 말로 풀어쓸 것
   - 긴 대시 `—`, 중점 `·`, 말줄임 `…` 금지. 하이픈 `-` 도 글머리로 쓰지 말 것
   - 수평선 `━━━`, `───`, `====` 등 장식선 금지
   - 백틱 `` ` ``, 트리플 백틱 금지. 코드/경로/명령어는 그냥 공백 양옆에 두고 평문처럼 적기
   - 마크다운 헤딩 `#`, `##` 금지
   - 볼드/이탤릭 `**`, `*`, `_` 금지
   - 마크다운 링크 `[text](url)` 금지
   - URL은 맨 마지막 한 줄에 작업일지 GitHub blob URL 하나만 허용. 본문 중간에는 URL 쓰지 말 것
   - **커밋 해시 목록 금지**. 필요하면 "최신 커밋은 5800aea로 원격 반영 완료" 한 줄만

   ### 본문 구조 예시 (산문체, 여러 프로젝트 통합본)
   프로젝트가 여러 개면 단락을 프로젝트 단위로 나눠 쓰되, 헤딩·불릿 없이 평서문으로 자연스럽게 넘긴다. 하나의 프로젝트만 있는 날은 기존처럼 영역별(기능/개선/배포 등) 단락 구성.
   ```
   YYYY.MM.DD 작업일지

   오늘은 (하루 전체 맥락 한 단락. 여러 프로젝트를 오갔다면 그 흐름도 한두 문장으로.)

   더치페이 계산기 쪽에서는
   (해당 프로젝트에서 한 작업을 이어지는 문장으로)

   약먹자에서는
   (다음 프로젝트 작업)

   기타 설정과 도구 쪽 작업도 있었다.
   (텔레그램 봇, 클로드 설정, 스킬 수정, 메모리 업데이트 등 프로젝트 소속이 아닌 작업)

   학습과 상담도 몇 건 있었다.
   (질문/상담성 대화 중 의미 있는 것만 한두 문장)

   남은 작업은
   (짧게 한 단락 또는 한 줄)

   오늘 기준 주요 저장소의 최신 커밋은 <해시>이고 원격 반영까지 끝난 상태다.

   https://github.com/<owner>/<repo>/blob/main/docs/worklog/YYYY-MM-DD.md
   ```

   **중요**: 파일(`docs/worklog/YYYY-MM-DD.md`)에 저장되는 내용과 채팅 응답 본문은 **같은 산문체** 로 통일. 이전처럼 파일은 마크다운, 응답은 평문으로 이원화하지 말 것. 파일도 섹션 헤딩 없이 위의 산문 구조로 작성.
